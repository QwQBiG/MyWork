肯德基 POS 机收银管理系统 — 答辩与项目说明文档

本系统是基于 C++ 开发的轻量级 KFC POS
终端收银管理程序。项目采用面向对象思想进行多文件分模块设计，实现了:
食品菜单管理、购物车计费、防错输入过滤以及本地数据持久化功能。

1. 项目文件结构 (Project Structure)

项目遵循高内聚、低耦合的工程化样式，将声明与实现分离：

├── Food.h              # 数据实体层：定义 Food 结构体及分类枚举辅助函数
├── MenuManager.h       # 数据持久层：管理食品信息读写与增删改逻辑的声明
├── MenuManager.cpp     # 数据持久层：文件读写及菜单维护逻辑的具体实现
├── Cart.h              # 业务逻辑层：购物车数据管理与账单计算声明
├── Cart.cpp            # 业务逻辑层：金额统计、找零计算与小票格式化输出实现
├── main.cpp            # 表现层/交互控制：菜单交互界面与鲁棒性输入校验
└── menu.txt            # 数据存储文件：程序运行时自动生成/读取的文本文件

2. 核心技术亮点 (Key Technical Highlights)

代码质量和工程规范：

  - 多文件模块化工程结构： 将数据模型、业务逻辑和界面交互彻底解耦。避免了单文件项目的臃肿，
    使得系统结构清晰，便于后期功能扩展。
  - 异常输入拦截与流控制（鲁棒性）： 本系统在输入端通过 std::cin.clear() 与 std::cin.ignore()
    建立了防死循环机制。当用户输入字母等非数值类型或越界负数时，系统不会崩溃或死循环，
	而是平滑拦截并引导重新输入。
  - 标准库容器与自动文件持久化： 没有使用复杂的数据库依赖，而是利用标准库的 
    std::vector 和 std::pair 动态存储数据，并配合
    std::ifstream 和 std::ofstream 实现食品菜单的自动载入与实时更新保存。

4. Q&A

Q1：为什么你的系统可以防止非数字输入引起的“死循环”？

回答： 标准的 std::cin 如果遇到类型不匹配的输入（例如让输入 int 却输入了字符串 abc），
       输入流会进入 fail 状态，后续的输入都会被忽略，导致程序陷入死循环。 
	   在我的项目中，我设计了 getSafeInt 和 getSafeDouble 辅助函数。每次读取失败时，
	   都会调用 std::cin.clear() 重置流状态，并使用 std::cin.ignore()清除输入缓冲区中的
	   无效残留字符，确保了系统的健壮性。

Q2：你的数据读写中，如何防止重名或重复食品号冲突？

回答： 在 MenuManager::addFood 方法执行之前，会先调用内部的 findFood(id)
	   函数。通过遍历现有的食品列表检测食品号是否已经存在。若已存在，则该方法会直接返回
	   false 并拦截添加操作，保证了食品号的唯一性。

Q3：既然要面向对象设计，为什么不用继承和多态（比如让“汉堡”和“饮料”继承自“食品”）？

回答：因为饮料、汉堡、小食品在底层的业务属性上是完全一致的（都具备编号、名称、单价以及分类）。
      为了避免过度设计（Over-engineering），保持代码的简洁与高效，
	  采用了基于实体属性分类（用 categoryId 进行属性区分）的设计。
	  这种扁平化的结构更易于进行文本文件的解析和单表存储，
	  同时也完全能满足当前肯德基 POS 系统的所有分类统计和输出需求。
